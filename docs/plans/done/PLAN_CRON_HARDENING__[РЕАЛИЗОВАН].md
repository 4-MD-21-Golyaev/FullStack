# PLAN_CRON_HARDENING

## Проблема

В `src/instrumentation.ts` cron-задачи оформлены неравномерно: для части задач есть `try/catch`, а для обработки outbox-задачи защитный контур отсутствует. Это снижает устойчивость фоновой обработки и затрудняет диагностику ошибок.

## Цель

Сделать фоновое выполнение cron-задач единообразным и безопасным:

- каждая задача должна иметь контролируемую обработку ошибок
- ошибки должны логироваться предсказуемо и информативно
- сбой одной задачи не должен делать поведение планировщика непрозрачным

## Логика исправления

### 1. Принять единый operational-контракт для всех cron jobs

Нужно зафиксировать, что каждая cron-задача:

- запускается из изолированного callback
- оборачивается в `try/catch`
- логирует контекст задачи и текст ошибки
- не скрывает ошибку молча

Причина:

- фоновая обработка должна быть наблюдаемой
- единый шаблон уменьшает шанс, что одна задача будет "особенной" и менее надежной

### 2. Выравнять структуру всех трех задач

Логически все зарегистрированные cron jobs должны быть оформлены одинаково:

- начало выполнения
- вызов use case
- обработка исключений
- при необходимости лог результата

Причина:

- одинаковая форма кода облегчает сопровождение
- проще находить различия по существу, а не по стилю реализации

### 3. Усилить логи контекстом задачи

Каждая ошибка должна быть связана с конкретной задачей:

- payment timeout
- order timeout
- outbox processing

Причина:

- иначе по логам трудно понять, какой именно контур упал
- это особенно важно, когда несколько задач запускаются каждую минуту

### 4. Не переносить бизнес-ошибки из use case наверх как "фатальные"

Cron-слой должен фиксировать сбой и завершать текущую итерацию, но не усложнять бизнес-логику самих use case.

Причина:

- use case остаются ответственными за свою предметную логику
- cron-обвязка отвечает только за orchestration и observability

### 5. Проверить единообразие поведения при повторных сбоях

Нужно, чтобы при периодических ошибках поведение оставалось контролируемым:

- задача не "замолкает"
- задача не генерирует неожиданные необработанные исключения
- планировщик продолжает вызывать следующие итерации

Причина:

- фоновая задача обычно должна быть "падающей безопасно", а не "падающей навсегда"

## Ожидаемый результат

После исправления все cron-задачи работают по одному шаблону наблюдаемости и отказоустойчивости. Ошибки в outbox, таймаутах или интеграциях не теряются и не выглядят как случайные необработанные исключения без контекста.

## Критерии проверки

### Функциональные

1. Все зарегистрированные cron-задачи продолжают запускаться после рефакторинга.
2. Сбой outbox-задачи не приводит к необработанному исключению без контекстного логирования.
3. Поведение payment timeout и order timeout не деградирует.
4. Успешные сценарии выполнения cron-задач не ломаются.

### Архитектурные

1. `src/instrumentation.ts` остается orchestration-слоем, а не местом для бизнес-логики.
2. Все cron-callback'и оформлены единообразно.
3. Изменение не требует переноса бизнес-логики из use case в cron-обвязку.

### По наблюдаемости

1. Для каждой задачи можно понять по логам, что именно выполнялось.
2. Для каждой ошибки видно, какой job-контур ее породил.
3. Ошибки не пропадают молча.

### По надежности

1. Периодический сбой внешней интеграции не ломает общий runtime-процесс cron-обработки.
2. Следующие итерации задач продолжают выполняться после ошибочной итерации.
3. Outbox-задача больше не является особым слабым местом по сравнению с двумя другими cron jobs.

### По тестированию

1. Есть проверка, что callback outbox-задачи безопасно обрабатывает исключение.
2. Есть проверка, что при ошибке выполняется логирование с контекстом задачи.
3. Есть проверка, что успешный сценарий работы cron-задач сохраняется.

## Практический сценарий приемки

1. Замокать ошибку в `ProcessOutboxUseCase`.
2. Запустить cron-контур или изолированный callback этой задачи.
3. Убедиться, что ошибка логируется с понятным префиксом.
4. Убедиться, что исключение не остается необработанным.
5. Повторить запуск и проверить, что задача продолжает вызываться на следующих итерациях.
