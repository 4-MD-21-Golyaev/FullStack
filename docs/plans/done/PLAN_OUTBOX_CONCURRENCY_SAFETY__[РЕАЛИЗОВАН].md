# PLAN_OUTBOX_CONCURRENCY_SAFETY

## Проблема

Текущая обработка outbox-событий выбирает pending events обычным чтением, а затем отдельно помечает их обработанными или ошибочными. При одновременном запуске нескольких воркеров один и тот же event может быть считан и обработан параллельно.

Это создает риск дублированной интеграционной отправки, особенно если одновременно работают cron и внутренний job endpoint.

## Цель

Сделать обработку outbox конкурентно-безопасной:

- одно событие должно забираться только одним воркером на обработку
- параллельные запуски не должны приводить к дублированной отправке
- механизм должен быть устойчивым к повторным запускам и временным ошибкам

## Логика исправления

### 1. Явно ввести стадию claim перед обработкой события

Сейчас событие сначала читается, а потом обрабатывается. Между этими шагами нет защиты от второго потребителя.

Нужно добавить логический этап:

- событие найдено
- событие атомарно "захвачено" текущим воркером
- только после этого начинается обработка

Причина:

- конкурентная безопасность в outbox обычно строится не на простом чтении, а на claim/lock-механизме

### 2. Разделить состояния события "pending" и "processing"

Если в модели есть только `pending` и `processed/failed`, воркеру трудно зарезервировать запись за собой без гонок.

Логически стоит выделить промежуточное состояние:

- `pending`
- `processing`
- `processed`
- `failed`

или эквивалентный механизм claim timestamp / owner marker.

Причина:

- это упрощает предотвращение двойной обработки
- делает состояние event более прозрачным для диагностики

### 3. Сделать выборку и захват события атомарными

Ключевое изменение не в том, чтобы "проверять внимательнее", а в том, чтобы не оставлять окно гонки между чтением и резервированием.

Причина:

- два воркера, читающие одни и те же pending events, иначе продолжат конфликтовать

### 4. Продумать recovery для зависших `processing` событий

Если воркер захватил событие и умер посередине обработки, событие не должно застрять навсегда в промежуточном состоянии.

Нужно предусмотреть:

- таймаут или lease-модель
- повторный возврат в pending при устаревшем claim
- либо другую форму безопасного переизъятия

Причина:

- защита от дублей не должна превращаться в риск вечной потери события

### 5. Сохранить текущую retry-логику, но встроить ее в новую модель

Сейчас есть `retryCount` и бизнес-разделение между временной и постоянной ошибкой. Это полезно сохранить.

Причина:

- задача не в том, чтобы заменить retry-механику, а в том, чтобы сделать ее совместимой с конкурентной обработкой

### 6. Согласовать cron и internal job endpoint с новой моделью

После изменения оба способа запуска должны использовать один и тот же безопасный механизм выдачи событий.

Причина:

- иначе один контур будет безопасным, а второй продолжит обходить защиту

## Ожидаемый результат

После исправления параллельные запуски outbox processing не приводят к двойной отправке одного и того же события. При этом события не застревают навсегда и сохраняют retry/recovery-поведение.

## Критерии проверки

### Функциональные

1. Одно pending-событие не обрабатывается двумя воркерами одновременно.
2. Успешно обработанное событие помечается завершенным один раз.
3. Ошибочное событие продолжает подчиняться retry-правилам.
4. Постоянная бизнес-ошибка по-прежнему может переводить событие в failed.

### Архитектурные

1. Механизм конкурентной безопасности реализован на уровне outbox persistence/use case, а не в обходных if-проверках на уровне route.
2. Cron и internal job endpoint используют одну и ту же модель захвата событий.
3. Retry и failure semantics не теряются при добавлении claim-логики.

### По надежности

1. Параллельный запуск двух воркеров не приводит к двойной отправке одного и того же интеграционного сообщения.
2. Если воркер падает после захвата события, существует предусмотренный сценарий его повторного возврата в обработку.
3. События не застревают бесконечно в промежуточном состоянии без recovery-механизма.

### По наблюдаемости

1. По состоянию записи можно понять, что событие ожидает обработки, обрабатывается, завершено или упало.
2. При сбоях claim/retry жизненный цикл события остается диагностируемым.

### По тестированию

1. Есть тест на конкурирующую обработку одного и того же события двумя воркерами.
2. Есть тест, подтверждающий, что только один воркер получает право обработать событие.
3. Есть тест на recovery зависшего `processing`-события.
4. Есть тест на сохранение retry-поведения после внедрения claim-механизма.

## Практический сценарий приемки

1. Создать одно pending outbox-событие.
2. Смоделировать два параллельных обработчика.
3. Запустить обработку одновременно.
4. Убедиться, что внешняя интеграция вызывается только один раз для этого события.
5. Смоделировать падение воркера после захвата события.
6. Убедиться, что через предусмотренный recovery-механизм событие снова становится доступным для обработки.
