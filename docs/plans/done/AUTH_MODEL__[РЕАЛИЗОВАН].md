# Модель аутентификации

## Способ аутентификации

Беспарольная аутентификация по одноразовому коду (OTP), отправляемому на email.

## Участники

| Сущность | Описание |
|----------|----------|
| User | Зарегистрированный пользователь с ролью (CUSTOMER, STAFF, ADMIN) |
| OTP | Одноразовый 6-значный код, привязанный к email, срок жизни 10 минут |
| Access Token | Краткоживущий JWT (15 мин), содержит идентификатор, роль и email пользователя |
| Refresh Token | Долгоживущий JWT (7 дней), хранится в БД, содержит только идентификатор пользователя |

## Жизненный цикл сессии

```
Регистрация → Запрос OTP → Ввод OTP → Выдача токенов → Активная сессия → Выход
```

### 1. Регистрация

Пользователь предоставляет email, телефон и адрес. Создаётся учётная запись с ролью CUSTOMER. Автоматический вход не происходит.

### 2. Вход

Пользователь запрашивает OTP на свой email. Код действителен 10 минут и может быть использован однократно. При успешной верификации кода выдаются оба токена.

### 3. Поддержание сессии

Access token используется для авторизации каждого запроса. По истечении 15 минут клиент обменивает refresh token на новую пару токенов (rotation — старый refresh token инвалидируется, выдаётся новый).

### 4. Выход

При выходе refresh token отзывается в БД, оба токена удаляются на стороне клиента.

## Двухуровневая токенная схема

```
┌─────────────────────────────────────────────────┐
│                  Access Token                    │
│  Назначение: авторизация запросов                │
│  Время жизни: 15 минут                           │
│  Хранение: httpOnly cookie (клиент)              │
│  Содержимое: userId, role, email                 │
│  Отзыв: невозможен (короткий TTL)               │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│                 Refresh Token                    │
│  Назначение: обновление access token             │
│  Время жизни: 7 дней                             │
│  Хранение: httpOnly cookie (клиент) + БД         │
│  Содержимое: userId                              │
│  Отзыв: мгновенный (флаг revoked в БД)          │
│  Rotation: при каждом обновлении выдаётся новый  │
└─────────────────────────────────────────────────┘
```

## Сценарии обновления и отзыва

| Сценарий | Поведение |
|----------|-----------|
| Access token истёк | Клиент вызывает refresh, получает новую пару токенов |
| Refresh token истёк | Сессия завершена, требуется повторный вход через OTP |
| Refresh token отозван | Сессия завершена, требуется повторный вход через OTP |
| Выход пользователя | Refresh token отзывается в БД, обе cookie удаляются |
| Компрометация access token | Злоумышленник имеет доступ максимум 15 минут |
| Компрометация refresh token | Администратор может мгновенно отозвать токен в БД |

## Модель авторизации

Ролевая модель (RBAC) с тремя ролями:

| Роль | Права |
|------|-------|
| CUSTOMER | Управление корзиной, создание и оплата заказов, просмотр своих заказов |
| STAFF | Права CUSTOMER + управление сборкой и доставкой заказов |
| ADMIN | Полный доступ |

Роль хранится в access token и проверяется на уровне middleware при каждом запросе. Подмена роли невозможна без знания секретного ключа подписи.

## Защита от угроз

| Угроза | Механизм защиты |
|--------|-----------------|
| XSS (кража токена из JS) | httpOnly cookie — токен недоступен из JavaScript |
| CSRF (запрос от чужого сайта) | sameSite cookie — браузер не отправляет cookie при cross-origin запросах |
| Перехват трафика (MITM) | secure cookie — передача только по HTTPS |
| Подделка токена | HMAC-SHA256 подпись — модификация payload без секрета невозможна |
| Утечка access token | Короткий TTL (15 мин) ограничивает окно уязвимости |
| Утечка refresh token | Хранение в БД позволяет мгновенный отзыв |
| Повторное использование refresh token | Rotation — использованный токен инвалидируется |
| Подбор OTP | 6-значный код + TTL 10 мин + одноразовое использование |

## Инварианты

1. Сессия невозможна без подтверждённого email (OTP).
2. Access token не хранится на сервере и не может быть отозван до истечения.
3. Refresh token всегда имеет запись в БД; без неё обновление невозможно.
4. Каждое обновление инвалидирует предыдущий refresh token (rotation).
5. Одному пользователю может соответствовать несколько активных refresh token (разные устройства).
6. Роль пользователя фиксируется в access token на момент выдачи; смена роли вступает в силу при следующем обновлении токена.
