# Материалы о работе с Claude Code для пояснительной записки

## Статус документа

Это расширенная аналитическая справка, собранная по материалам репозитория. Ее задача не в кратком пересказе, а в накоплении максимума полезной фактуры в одном месте. Документ специально сделан избыточным: из него должно быть удобно извлекать как короткие абзацы для основной части ВКР, так и более подробные рассуждения для разделов про процесс разработки, ограничения применения ИИ и организацию инженерной работы.

Структура документа построена так, чтобы последовательно ответить на пять вопросов:

1. Есть ли в проекте прямые признаки использования `Claude Code`.
2. Как именно была организована работа с агентом.
3. Как этот процесс менялся во времени.
4. Какой вклад агента виден в итоговом результате проекта.
5. Какие ограничения и риски такого подхода были обнаружены и как они компенсировались.

## 1. Общий тезис

По репозиторию можно уверенно утверждать, что `Claude Code` использовался в проекте системно. Это подтверждается не одним косвенным признаком, а целым набором независимых источников внутри проекта:

- прямое утверждение в публичном описании проекта;
- наличие отдельного регламента `CLAUDE.md`;
- наличие каталога `.claude` с внутренними правилами, настройками и рабочими деревьями;
- специальный workflow для задач разного типа;
- отдельное архитектурное решение о независимом review-агенте;
- история коммитов, где видны как крупные функциональные изменения, так и усиление правил, тестирования и аудита;
- набор планов, аудитов и решений, фиксирующих практику взаимодействия с агентом.

Из этого следует важный для ВКР вывод: в данном проекте `Claude Code` был не просто внешним помощником для генерации кода, а частью управляемого процесса разработки.

## 2. Что в проекте прямо подтверждает использование Claude Code

### 2.1. Прямое утверждение в README

В `README.md` проект описан как eGrocery-платформа, разработанная в рамках ВКР. В этом же документе есть прямое утверждение о том, что весь код проекта разработан с использованием возможностей ИИ-агента `Claude Code (Anthropic)`.

Для пояснительной записки это особенно ценно по двум причинам:

- утверждение находится в публичном описании проекта, а не в черновой внутренней заметке;
- формулировка относится ко всему проекту, а не к отдельным его частям.

Это означает, что автор проекта сознательно выносит использование Claude Code в число характеристик процесса разработки.

### 2.2. Наличие отдельного файла `CLAUDE.md`

В корне проекта находится файл `CLAUDE.md`, который объявлен как руководство для `Claude Code` при работе с этим репозиторием.

Сам факт существования такого файла уже говорит о многом:

- агент использовался не разово, а регулярно;
- под проект была подготовлена специальная инструкция;
- разработчик не полагался только на общий интерфейс агента, а адаптировал его поведение под архитектуру, стек и организацию кода.

Содержательно `CLAUDE.md` включает:

- команды запуска и сборки проекта;
- переменные окружения;
- описание архитектуры;
- правила импорта и слоев;
- правила делегирования задач агентам;
- условия, при которых задача считается завершенной.

Таким образом, `CLAUDE.md` выступает как операционный контракт между проектом и ИИ-агентом.

### 2.3. Наличие каталога `.claude`

В проекте присутствует каталог `.claude`, внутри которого находятся:

- `.claude/rules/` с набором специализированных правил;
- `.claude/settings.local.json`;
- `.claude/worktrees/` с отдельными рабочими деревьями.

Если бы Claude Code использовался только эпизодически, такой инфраструктуры, скорее всего, не потребовалось бы. Наличие отдельного каталога с правилами и `worktrees` говорит о том, что агентная разработка была встроена в повседневную работу над проектом.

### 2.4. Наличие нескольких `worktree`

Внутри `.claude/worktrees` присутствуют отдельные рабочие директории:

- `agent-a1f084f1`
- `agent-a3eee6be`
- `agent-a4d1e99d`

В локальном Git им соответствуют ветки `worktree-agent-*`.

Из этого нельзя механически вывести, какие именно задачи решались в каждом дереве и как долго они существовали. Но в совокупности с правилами workflow это является сильным косвенным доказательством того, что проект был организован под многoагентную или параллельную работу.

### 2.5. Следы агентной логики в самих правилах

Проект содержит не просто общие заметки об ИИ, а формализованные инструкции, прямо построенные вокруг понятия агента:

- implementation agent;
- review agent;
- test+review agent;
- mixed tasks;
- parallel execution;
- plan gate;
- post-completion checks.

Это важно, потому что подтверждает: агент мыслился не как "умная автодополнение", а как отдельная операционная единица процесса разработки.

## 4. Как был организован процесс работы с Claude Code

### 4.1. Принцип read-first

Один из базовых пунктов workflow требует, чтобы перед модификацией любого файла агент сначала прочитал сам файл и его прямые зависимости. Для UI отдельно оговорено, что перед реализацией нужно читать `src/shared/ui/index.ts`.

Практический смысл этого правила:

- снизить вероятность того, что агент дублирует уже существующие компоненты или паттерны;
- принудительно погрузить агента в локальный контекст проекта;
- уменьшить риск "генерации в вакууме", когда модель выдает правдоподобный, но плохо интегрированный код.

Для ВКР это хороший аргумент в пользу того, что результат работы с ИИ зависел не только от модели, но и от качества локального контекста, который ей предъявлялся.

### 4.2. Принцип scope lock

Workflow запрещает агенту расширять задачу за пределы запроса пользователя. Это особенно интересно в контексте LLM, потому что модели часто склонны "додумывать" улучшения, рефакторинги и дополнительные изменения.

С инженерной точки зрения такое правило нужно для двух вещей:

- удерживать результат в предсказуемых границах;
- не смешивать выполнение задачи с самопроизвольным рефакторингом.

Иначе говоря, разработчик ограничивал не только содержимое кода, но и "поведенческую свободу" агента.

### 4.3. Plan gate

Если задача затрагивает несколько файлов или несколько архитектурных слоев, workflow требует сначала построить краткий план. Это интересный момент для ВКР, потому что показывает: агент применялся не как безусловный исполнитель, а как участник процесса, в котором сначала фиксируется намерение, а потом уже выполняется работа.

Plan gate полезен как минимум в трех смыслах:

- снижает риск ошибочного вмешательства в соседние части системы;
- помогает разделить задачу на подзадачи;
- облегчает последующее делегирование разным агентам.

### 4.4. Разделение frontend и backend по разным стратегиям

Проект различает frontend- и backend-изменения не только по расположению файлов, но и по организации работы с агентом.

Для UI:

- работа всегда идет через отдельного implementation-agent;
- затем обязателен отдельный review-agent;
- даже минимальное изменение требует review;
- проверяется не только типизация, но и композиция интерфейса, placement, barrel-export, стили и соответствие FSD-правилам.

Для backend:

- реализация может быть прямой только для небольших изменений;
- но test+review обязателен всегда;
- особое внимание уделяется доменным инвариантам, переходам состояний, транзакциям и edge cases.

Такое разделение отражает здравую инженерную гипотезу: UI-чувствителен к семантическим и композиционным ошибкам, backend - к инвариантам, транзакционности и корректности тестового покрытия.

### 4.5. Формальная обработка ошибок

Workflow отдельно оговаривает: если агент получил ошибку, тест упал или `tsc` выдал проблему, нужно читать вывод полностью, исправлять корневую причину и заново запускать именно ту проверку, которая не прошла.

Это важно, потому что одна из типичных проблем при работе с ИИ - склонность к поверхностному устранению симптомов. В проекте этому противопоставляется формальное правило: не игнорировать ошибку, не обходить ее и не считать задачу закрытой без повторной проверки.

### 4.6. Постусловия завершения

После завершения работ требуется обязательный `tsc --noEmit`. Для backend также явно описана необходимость прохождения тестов.

Это простой, но показательный момент: проект не доверяет агенту "на слово". Даже если код выглядит правдоподобно, он должен пройти формальную техническую проверку.

## 5. Детальный разбор UI-процесса

### 5.1. Почему UI выделен в отдельную категорию

Фронтенд в проекте не рассматривается как "менее критичная часть". Наоборот, именно для него введены самые строгие ограничения. Это означает, что в реальной работе оказалось недостаточно просто получать работающий JSX. Требовалось, чтобы он:

- соответствовал уже существующей библиотеке shared/ui;
- вписывался в FSD-структуру;
- соблюдал систему токенов;
- не вносил стилистический хаос;
- не подрывал единообразие интерфейса.

### 5.2. Двухфазный pipeline

В правилах UI явно зафиксированы две обязательные фазы:

1. Implementation agent.
2. Review agent.

Это можно интерпретировать как разделение ролей между "автором" и "контролером". Важно, что review вынесен в отдельного агента, а не сводится к self-check того же самого контекста.

### 5.3. Что проверял implementation-agent

Implementation-agent должен был:

- прочитать существующие shared/ui-компоненты;
- прочитать правила композиции;
- не использовать raw HTML там, где существует компонент-замена;
- класть новый компонент в правильную группу;
- экспортировать его через barrel;
- использовать CSS tokens;
- выполнять предварительную самопроверку;
- запускать TypeScript-проверку.

То есть даже фаза "реализации" уже включала внутренний контроль.

### 5.4. Что проверял review-agent

Review-agent проверял:

- нет ли запрещенного raw HTML;
- есть ли экспорт из barrel;
- лежит ли компонент в правильной директории;
- нет ли запрещенных CSS-паттернов;
- корректны ли импорты внутри shared/ui;
- соответствует ли реализация требованиям задачи;
- соблюдаются ли FSD-правила;
- обработаны ли edge cases;
- совпадает ли решение с существующими паттернами.

Это уже не просто линтер и не просто type-check. Это попытка формализовать качественное, архитектурное ревью в терминах, доступных агенту.

### 5.5. Почему это важно для ВКР

В пояснительной записке можно показать, что опыт проекта привел к важному выводу: при работе с ИИ особенно полезны те ограничения, которые можно выразить в виде проверяемых правил. Там, где правила конкретны и бинарны, качество результата заметно выше. Там, где проверка опирается на "общий здравый смысл", возрастает риск пропуска дефектов.

## 6. Детальный разбор backend-процесса

### 6.1. Backend как зона доменных инвариантов

Судя по `testing.md` и общей архитектуре проекта, backend рассматривался как область, где особенно критичны:

- состояния заказа;
- корректность переходов;
- транзакционные границы;
- идемпотентность;
- правила взаимодействия с внешними системами;
- race conditions;
- разграничение ответственности по слоям.

То есть к backend-задачам предъявлялись не только требования "чтобы работало", но и требования "чтобы сохранялись инварианты предметной области".

### 6.2. Почему обязательны тесты

Правила backend-тестирования требуют покрывать:

- успешные пути;
- ошибочные пути;
- пограничные случаи;
- новые переходы;
- новые repository methods;
- API-ошибки и ответы авторизации.

Это особенно важно в проекте с state machine и интеграциями, где дефекты могут проявляться не в простой форме "страница не рендерится", а в виде тихого нарушения инвариантов.

### 6.3. Что именно проверял review на backend

В правилах review отдельно упомянуты:

- layer compliance;
- domain invariants;
- transaction boundaries;
- edge cases;
- consistency.

Это значит, что backend-review был настроен не только на unit-level корректность, но и на архитектурную дисциплину.

### 6.4. Вывод для ВКР

На материале проекта можно показать, что ИИ особенно полезен там, где задача уже хорошо формализована, но его результат должен обязательно проверяться на уровне доменной модели и архитектуры. Для backend это означает: LLM ускоряет реализацию, но проверка инвариантов не может быть опущена.

## 7. Почему понадобился независимый review-agent: подробный разбор ADR-001

### 7.1. Контекст решения

ADR-001 посвящен UI-компонентам. Из документа видно, что на момент его принятия в проекте уже существовала библиотека shared/ui с жесткими правилами композиции. Это означает, что проблема была не в отсутствии стандартов, а в отсутствии гарантии их соблюдения агентом.

### 7.2. Зафиксированные типы ошибок

В ADR перечислены типовые нарушения, которые делали агентный вывод недостаточно надежным:

- использование raw HTML вместо существующих проектных примитивов;
- хардкод CSS-значений;
- пропуск barrel-export;
- нарушение организационных правил компонента.

Эти ошибки особенно интересны, потому что они не всегда ломают сборку и не всегда легко обнаруживаются автоматикой. То есть проблема была не в синтаксисе, а в семантическом качестве интеграции.

### 7.3. Ключевая мысль ADR

Одна из самых важных идей ADR формулируется так: документация задает стандарт, но не гарантирует его выполнение. Это очень сильный исследовательский вывод для ВКР.

Он означает, что при использовании ИИ нужно различать:

- наличие правил;
- соблюдение правил;
- механизм проверки соблюдения правил.

Если есть только первое, качество остается нестабильным.

### 7.4. Почему отдельный агент лучше self-review

ADR исходит из того, что self-review в том же контексте недостаточно надежен. Причина понятна: агент уже "вложился" в конкретное решение и склонен считать его правдоподобным. Отдельный review-agent, читающий только результат, получает более независимую позицию.

Это можно интерпретировать как практическую борьбу с confirmation bias в генеративной разработке.

### 7.5. Итоговая организационная схема

Решение, принятое в ADR, фактически превращает review в обязательную отдельную стадию жизненного цикла UI-задачи. Таким образом, процесс разработки UI становится ближе к схеме:

1. постановка задачи;
2. реализация;
3. независимая проверка;
4. исправление найденных нарушений;
5. повторная проверка.

Именно этот момент делает опыт проекта ценным для описания в пояснительной записке: использование ИИ не сократило инженерный процесс до "одного запроса", а наоборот, заставило явно оформить стадии контроля.

## 12. Ограничения и риски использования Claude Code

Ниже собраны основные риски, которые можно извлечь из проекта.

### 12.1. Риск нарушения локальных архитектурных правил

Самый явно зафиксированный риск - агент может сгенерировать код, который формально работает, но не соответствует локальным архитектурным соглашениям проекта.

В случае этого проекта такие нарушения проявлялись как:

- raw HTML вместо существующих UI-компонентов;
- хардкод стилей;
- неправильное размещение компонентов;
- отсутствие barrel export;
- расхождение с FSD-структурой.

### 12.2. Риск confirmation bias при self-review

Если тот же самый агент, который написал решение, сразу же сам его оценивает в том же контексте, повышается вероятность, что он не заметит собственные ошибки или будет склонен трактовать спорные решения в свою пользу.

Именно поэтому был введен независимый review-agent.

### 12.3. Риск недостаточного тестового покрытия

Для backend-проектов с доменными инвариантами и интеграциями особенно опасен сценарий, когда код выглядит логичным, но не покрывает пограничные случаи.

Проект реагирует на это обязательными test+review-правилами и расширением покрытия, что отражено и в аудите.

### 12.4. Риск скрытых семантических ошибок

Одна из главных сложностей с LLM состоит в том, что они могут генерировать код, который:

- типизируется;
- компилируется;
- выглядит убедительно;
- но нарушает инвариант или организационное правило.

Именно этот тип риска лучше всего иллюстрируется на примере UI-композиции и доменных ограничений backend.

### 12.5. Риск роста стоимости контроля качества

Чем активнее используется агент, тем больше нужно:

- правил;
- review;
- тестов;
- планов;
- аудитов;
- повторных проверок.

То есть ИИ экономит время на части реализации, но одновременно порождает дополнительную работу по верификации результата.

| Ошибка маскируется поверхностным исправлением | тесты или type-check падают повторно | `error protocol` |
| Self-review недостаточно надежен | confirmation bias, пропуск собственных ошибок | независимый review-agent |
| Backend-код остается без покрытия | пропуск edge cases и инвариантов | `test+review`, обязательные тесты |
| Формально корректный код нарушает архитектуру | слой импортирует лишнее, ломаются правила транзакций | backend review по layer compliance и invariants |
| Рост сложности проекта снижает надежность ручного контроля | скрытые дефекты, неочевидные расхождения | аудит, планы, статистика покрытия |

## 15. Что лучше формулировать осторожно

Есть и такие утверждения, которые лучше не подавать как жестко доказанные:

- что каждый конкретный коммит полностью сделан агентом;
- что все архитектурные документы написаны агентом без участия человека;
- что все `worktree` использовались одновременно;
- что параллельные агенты реально запускались в каждом случае mixed tasks, а не просто были предусмотрены правилами.

Корректнее писать так:

"Структура репозитория, правила workflow и наличие отдельных `worktree` указывают на подготовленность проекта к системному и частично параллельному использованию `Claude Code`."

## 16. Готовые развернутые формулировки для пояснительной записки

### 16.1. Формулировка о роли Claude Code

В ходе выполнения выпускной квалификационной работы при разработке программной системы использовался ИИ-агент `Claude Code`. Его роль не сводилась к генерации отдельных фрагментов исходного кода. В проекте была создана специальная среда агентной разработки, включавшая отдельный регламент работы, набор проектно-специфических правил, механизмы независимой проверки результатов, а также обязательные процедуры тестирования и верификации. Тем самым ИИ-агент выступал как встроенный инструмент инженерной разработки, а не как внешний чат-сервис для получения разовых подсказок.

### 16.2. Формулировка об организации процесса

Работа с `Claude Code` была организована как регламентированный процесс. Перед внесением изменений агент должен был изучать модифицируемые файлы и их зависимости, а при работе с комплексными задачами требовалось предварительное планирование. Для UI-компонентов использовалась двухэтапная схема "агент реализации - независимый агент проверки", а для backend-изменений обязательными были написание тестов и review с проверкой доменных инвариантов, архитектурных ограничений и пограничных сценариев. Такой подход позволил использовать ИИ-агента в составе контролируемого инженерного контура.

### 16.3. Формулировка об ограничениях

Практическое использование `Claude Code` показало, что наличие даже подробной документации не гарантирует автоматическое соблюдение архитектурных и стилевых правил проекта. В процессе разработки было зафиксировано, что агент при создании UI-компонентов мог нарушать ограничения композиции, использовать raw HTML вместо проектных примитивов и хардкодить стилевые значения. Для снижения этих рисков был введен независимый review-agent с формализованным набором критериев проверки. Это увеличило трудоемкость процесса, однако позволило повысить надежность и воспроизводимость результатов.

### 16.4. Формулировка о вкладе в результат

По структуре репозитория, истории изменений и проектной документации видно, что с использованием `Claude Code` создавались как базовые механизмы предметной области, так и прикладные и интерфейсные части системы. В частности, в проекте реализованы жизненный цикл заказа, платежная логика, корзина, личный кабинет, ролевая модель доступа, интеграции с внешними сервисами, а также интерфейсы для администратора, сборщика и курьера. Следовательно, ИИ-агент применялся при разработке значительной части функциональности программной системы.

### 16.5. Формулировка о научно-практическом выводе

Опыт разработки показал, что наибольшую эффективность ИИ-агент демонстрирует не при свободной генерации кода, а при включении в формализованный инженерный процесс, где заранее заданы контекст, границы задачи, критерии качества и процедуры проверки. В этом случае ИИ способен существенно ускорять реализацию, однако его применение требует дополнительных организационных мер, направленных на предотвращение архитектурных, семантических и интеграционных ошибок.
