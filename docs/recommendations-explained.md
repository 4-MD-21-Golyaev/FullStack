# Как работает система рекомендаций

## Архитектура: три уровня данных

Система не хранит готовые рекомендации заранее — она вычисляет их **в момент запроса** из истории заказов. Источник истины — таблицы `Order` + `OrderItem`, отфильтрованные по `OrderStatus.code IN ('DELIVERED', 'CLOSED')` (только реально завершённые покупки, отменённые игнорируются).

```
PostgreSQL ──→ $queryRaw агрегации ──→ use cases (с fallback) ──→ API ──→ React хуки
                  ↑ time-decay              ↑ personalized?         ↑ unstable_cache
                                                                      (только глобальные)
```

---

## Три типа рекомендаций

### 1. Топ категорий (главная страница)

**Запрос:** «Какие категории чаще всего покупают?»

```sql
SELECT p."categoryId", COUNT(DISTINCT o.id) AS order_count, ⟨score⟩
FROM OrderItem oi JOIN Order o ON … JOIN Product p ON …
WHERE OrderStatus.code IN ('DELIVERED','CLOSED')
GROUP BY p.categoryId
ORDER BY score DESC
```

Дальше in-memory **сворачиваем к корням** дерева категорий (`rootOnly: true` + `findRootCategoryId`), чтобы не получить три подкатегории «Молочное» в трёх слотах. Глобальная версия для гостей; для авторизованных — тот же запрос с `AND o.userId = $userId` (персональный affinity).

### 2. Топ товаров (каталог, фильтр категории)

То же самое, но `GROUP BY oi.productId`. Если задан `categoryId` — расширяется в `categoryIds[]` через BFS по дереву (`collectDescendantIds`), чтобы топ под «Бакалея» учитывал и «Бакалея/Крупы», и «Бакалея/Макароны».

### 3. «С этим товаром покупают» — item-to-item collaborative filtering

Это самый интересный кусок. Идея: «два товара похожи, если их часто покупают в одних и тех же заказах».

**Метрика — Jaccard similarity:**
```
              | заказы с A ∩ заказы с B |
J(A, B) = ─────────────────────────────────
              | заказы с A ∪ заказы с B |
```

В SQL это делается одним запросом с тремя CTE:

```sql
WITH target_orders AS (   -- заказы, где встречается seed-товар
    SELECT DISTINCT o.id FROM OrderItem ... WHERE productId = $seed
),
co AS (                   -- товары-соседи + размер пересечения (numerator)
    SELECT productId, COUNT(DISTINCT orderId) AS inter
    FROM OrderItem WHERE orderId IN target_orders AND productId <> $seed
    GROUP BY productId
),
candidate_orders AS (     -- сколько всего заказов у каждого соседа
    SELECT productId, COUNT(DISTINCT orderId) AS total ...
)
SELECT inter / NULLIF(target_count + total - inter, 0) AS jaccard_score
FROM co JOIN candidate_orders ON ...
WHERE inter >= 3          -- порог: нужно минимум 3 совместных заказа
ORDER BY jaccard_score DESC
```

Знаменатель — формула включения-исключения: `|A ∪ B| = |A| + |B| − |A ∩ B|`. `NULLIF(…, 0)` страхует от деления на 0 (никогда не должно произойти при `inter >= 3`, но pedantic safety).

**Cold-start товара:** если соседей < 3 — фолбэк на топ той же категории, исключая сам товар.

## Time-decay (опциональное улучшение)

Простой `COUNT(*)` относится одинаково к заказам годовой давности и вчерашним. Time-decay вместо счёта суммирует **взвешенные** вклады:

```sql
SUM(CASE
    WHEN o.createdAt > NOW() - INTERVAL '30 days' THEN 1.0
    WHEN o.createdAt > NOW() - INTERVAL '90 days' THEN 0.5
    ELSE 0.25
END)
```

Дискретные веса (а не экспонента `e^(-λt)`) выбраны намеренно: их легче защитить в ВКР («вот таблица весов») и не нужно подбирать гиперпараметр λ. Включён для категорий и продуктов, выключен для CF (там стабильность связей важнее свежести).

## Fallback-цепочки (cold-start защита)

Каждый use case умеет деградировать — клиент всегда получает непустой результат, если в БД хоть что-то есть:

| Сценарий | Что происходит | Что в ответе |
|---|---|---|
| Гость на главной | global top categories | `personalized: false, fallbackUsed: false` |
| Auth, нет истории | global top categories | `personalized: false, fallbackUsed: true` |
| Auth, есть история | user-specific top | `personalized: true, fallbackUsed: false` |
| Товар без соседей | top той же категории, минус сам товар | `fallbackUsed: true` |
| Товар не найден | пустой массив | `fallbackUsed: true` |

Поле `fallbackUsed` в JSON — для отладки и метрик мониторинга, фронт его не показывает.

## Кеширование

- **Глобальные роуты** (`/top-categories`, `/related`) обёрнуты в `unstable_cache(revalidate: 600s, tag: 'recommendations:global')`. Запрос для всех гостей идёт в БД раз в 10 минут.
- **Персональные роуты** (`/categories`, `/products`) **не кешируются** — иначе нужен per-user ключ кеша, что усложняет инвалидацию. С индексами агрегация для одного пользователя даёт миллисекунды.
- **Инвалидация**: `CloseOrderUseCase` (закрытие заказа = новые подтверждённые покупки) вызывает `revalidateTag('recommendations:global', 'default')` — следующий запрос пересчитает.

# Метрики оценки

Скрипт `recommendations-eval.ts` делает **офлайн A/B-сравнение**: split-by-time (80% старых заказов = train, 20% новых = test), для каждого подхода для каждого test-пользователя берёт top-10 рекомендаций из train и сравнивает с тем, что пользователь реально купил в test.

## Что значит каждая метрика

### Precision@10 — «насколько чисты рекомендации»

```
Precision@K = |recommended ∩ test_purchased| / K
```

«Из 10 наших рекомендаций сколько процентов реально куплены». Высокая Precision = мы не тратим слоты на промахи.

В таблице **3.85% у global-popular-decay** означает: в среднем 0.4 из 10 рекомендованных товаров пользователь действительно купил в тестовом периоде. Кажется мало, но это нормально для retail с каталогом 3590 товаров и средним чеком 2-6 позиций.

### Recall@10 — «насколько полные рекомендации»

```
Recall@K = |recommended ∩ test_purchased| / |test_purchased|
```

«Из всего, что пользователь купил, сколько мы угадали». Низкий Recall означает: даже если мы попадаем — мы покрываем малую долю реальных покупок.

Recall всегда «штрафует» большие test-корзины: если человек купил 30 товаров, а мы рекомендуем 10, потолок Recall = 33%.

### Hit Rate@10 — «угадали хотя бы один товар»

```
HitRate = (число пользователей, у которых ≥1 попадание) / (всего пользователей)
```

Бинарная метрика на уровне пользователя. Полезна, когда даже одна правильная рекомендация = ценность (товар попал в карусель — пользователь кликнул).

**23.1% у decay** = почти каждый 4-й пользователь увидел в топ-10 что-то реально нужное.

### Coverage — «какую долю каталога мы вообще показываем»

```
Coverage = |уникальные товары в рекомендациях по всем юзерам| / |каталог|
```

Если Coverage 0.28% (как у global-popular) — система всем пихает один и тот же топ. Каталог из 3590 товаров используется на 10 SKU.

Coverage — это **антагонист точности**: показывая один и тот же популярный набор всем, легко поднять Hit Rate, но 99.7% каталога пылится. Random имеет Coverage 3.45% — в 12 раз больше товаров видны.

### Diversity — «насколько разнообразен один набор рекомендаций»

```
Diversity = avg(уникальных категорий в топ-K) / K
```

Берётся среднее по всем пользователям. Random `0.92` означает: его 10 рекомендаций раскиданы почти по 10 разным категориям. Global-popular `0.40` — все 10 укладываются в ~4 категории (что и ожидаемо: в топе по покупаемости обычно молочка + хлеб + бакалея).

Diversity важна для UX: пользователь не хочет видеть «варёная колбаса A, варёная колбаса B, варёная колбаса C» — это вызывает choice fatigue.

## Главный сюжет для ВКР

**Никакой подход не оптимален по всем метрикам сразу.** Это и есть основной нарратив главы:

- Хочешь точности → time-decay popularity (но низкое разнообразие, popularity bias)
- Хочешь покрыть каталог → CF (но ниже точность)
- Хочешь личного отклика → user-popular (но требует много истории на пользователя)
- Хочешь max diversity → random (но pointless)

В проде такие системы комбинируют: **гибрид** popularity + CF + персонализация + diversity-rerank. У нас в проекте используется простая, но осознанная стратегия: глобальный topic decay на главной, item-to-item на странице товара, user affinity при сортировке каталога. Каждый подход применён там, где его сильные стороны нужны, а слабости не критичны.
