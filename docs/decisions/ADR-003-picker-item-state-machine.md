# ADR-003: Picker item state machine и применение AbsenceResolutionStrategy

**Статус:** Принято
**Дата:** 2026-03-26

---

## Контекст

В домене существует `AbsenceResolutionStrategy` с четырьмя значениями (`AUTO_REMOVE`, `CALL_REMOVE`, `AUTO_REPLACE`, `CALL_REPLACE`), которая хранится в заказе, но нигде не влияла на логику приложения — ни в use cases, ни в UI. Пикер мог завершить сборку независимо от стратегии. Кроме того, в `UpdateOrderItemsUseCase` был запрет на qty = 0, что делало удаление позиции невозможным.

---

## Решение

### State machine для позиции заказа

Каждая позиция в процессе сборки имеет одно из четырёх состояний:

```
Необработано ──── найти (qty > 0) ────→ Собрано
     │                                      │
     └── отметить отсутствующим ──→ Отсутствует ←── убрать замену
                                       │    │
                              добавить │    └── восстановить → Необработано
                              замену   │
                                       ↓
                                  Заменено
```

| Состояние | Смысл |
|-----------|-------|
| **Необработано** | Пикер ещё не обработал позицию (qty = 0, не отмечена отсутствующей) |
| **Собрано** | Пикер нашёл товар, qty > 0 |
| **Отсутствует** | Пикер явно отметил позицию отсутствующей, замена не добавлена |
| **Заменено** | Позиция отсутствует, замена добавлена и связана в UI-state |

Ключевое разграничение: `qty = 0` в состоянии «Необработано» означает «ещё не дошёл», а не «не нашёл». Явное действие «Отсутствует» — первоклассная операция, отдельная от изменения количества.

### Как стратегия ограничивает действия

| Стратегия | Действие «Отсутствует» | Добавить замену | Завершить без замены |
|-----------|----------------------|----------------|---------------------|
| `AUTO_REMOVE` | ✅ | ❌ (покупатель запретил) | ✅ |
| `CALL_REMOVE` | ✅ | ✅ (мог передумать при звонке) | ✅ |
| `AUTO_REPLACE` | ✅ | ✅ (приоритет) | ✅ + предупреждение |
| `CALL_REPLACE` | ✅ | ✅ (приоритет) | ✅ + предупреждение |

**CALL_*** — показывает `tel:` кнопку в шапке заказа. Звонок не является обязательным технически; доверие пикеру.

### Правила завершения сборки

1. Есть позиции в состоянии **Необработано** → заблокировано
2. Нет ни одной позиции в **Собрано** или **Заменено** → заблокировано (пустой заказ)
3. ~~Стратегия REPLACE и есть позиции в **Отсутствует** → заблокировано~~ — **отменено** (см. корректировку ниже)

### Связь «удалённая позиция → замена»

Связь хранится только в клиентском состоянии (Map<productId, replacedByProductId | null>). Серверу она не нужна: `UpdateOrderItemsUseCase` принимает финальный список позиций, где замены уже включены как обычные items.

### Изменения в существующем коде

| Слой | Файл | Изменение |
|------|------|-----------|
| application | `UpdateOrderItemsUseCase` | Разрешить qty = 0; позиция с qty = 0 исключается из `newItems` |
| application | `CompletePickingUseCase` | Guards: нет Необработанных; не пустой заказ |
| UI | `PickingWorkspace` | Четыре состояния, группировка, кнопка «Отсутствует», restore, `tel:` при CALL_*, поиск замены, группа «Заменено» |
| feature | `features/product-search` | `useProductSearch` + `ProductSearchModal` — reusable поиск по каталогу |

## Корректировка первоначального решения (2026-03-26)

**Проблема с guard 3:** жёсткий блок «REPLACE + отсутствующие → нельзя завершить» ломает систему в случае, когда замены реально нет в наличии. Пикер оказывался в тупике: завершить нельзя, подходящей замены нет. Единственный выход — добавить любой товар «для галочки», что полностью обессмысливает guard.

**Проблема с AUTO_REMOVE + замена:** покупатель явно выбрал «удалить без звонка», разрешение замены противоречило его воле.

**Принятое решение:** guard 3 удалён с сервера. REPLACE-стратегии показывают информационное предупреждение в UI («N поз. без замены — будут удалены»), но не блокируют завершение. AUTO_REMOVE скрывает кнопку «Выбрать замену».

## Отложено

- Принудительное подтверждение звонка при CALL_*
- Бизнес-логика частичного отсутствия: `qty < maxQty` — пикер собрал меньше заказанного (Интерпретация B: стратегия применяется к непобранной части; требует нового поля в OrderItem и переработки state machine)
- Расширенная логика AUTO_REPLACE (каталог аналогов)

---

## Альтернативы, которые не были выбраны

**3 состояния (Необработано / Собрано / Отсутствует):** не разграничивает «ещё не обработал» и «явно отсутствует»; делает невозможным корректное применение REPLACE guard'а без отдельного поля-флага.

**Серверное хранение связи замены:** избыточно — сервер получает только финальный список items, история переходов UI не нужна на уровне доменной модели.
