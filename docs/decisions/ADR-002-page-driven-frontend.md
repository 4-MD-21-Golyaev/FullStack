# ADR-002: Page-Driven Development для реализации customer-фронтенда

**Статус:** Принято
**Дата:** 2026-03-22

---

## Контекст

Проект включает полный UI Kit из 86 компонентов (Figma: `eV3fLo7RMJcyNqnFeuwHmg`, страница `node-id=1-7`). Из них реализовано 30 к моменту принятия решения. Предстоит реализовать customer-контур (8 страниц) и закрыть ВКР.

Возможных подходов два:

**Подход A — UI Kit First:** сначала реализовать все оставшиеся 56 компонентов, затем собирать страницы.

**Подход B — Page-Driven Development:** строить страницы одну за другой по приоритету пользовательского пути; для каждой страницы реализовывать только компоненты, необходимые для её рендера.

Перед выбором было проведено картографирование зависимостей: для каждой из 8 customer-страниц построены транзитивные карты (страница → виджеты → молекулы → атомы), зафиксированные в `docs/plans/PAGE_QUEUE.md`.

---

## Решение

Принят **Page-Driven Development (Подход B)**.

Страницы реализуются в порядке пользовательского пути:

1. Главная `/`
2. Каталог `/catalog`
3. Карточка товара `/catalog/[slug]`
4. Корзина `/cart`
5. Чекаут `/checkout`
6. Спасибо `/checkout/success`
7. ЛК — история заказов `/account/orders`
8. ЛК — детали заказа `/account/orders/[id]`

Для каждой страницы: определяется минимальный набор нужных компонентов → реализуются только они (в порядке зависимостей: атомы → молекулы → виджеты) → собирается страница → переход к следующей.

### Итоговый объём (результат картографирования)

Из 86 компонентов UI Kit для покрытия всех 8 страниц нужны **26** (25 новых + 1 расширение существующего). Остальные 60 не входят в текущий объём.

| Категория | Кол-во |
|---|---|
| Новые `shared/ui` компоненты | 25 |
| Расширение существующего компонента | 1 |
| Новые виджеты | 6 |
| **Итого единиц работ** | **32** |

Расширяемый компонент: `OrderStatusBadge` → добавляются customer-состояния (В сборке / В пути / Доставлен / Отменен) из Figma `1097:18614`. Отдельный компонент не создаётся.

Полный реестр с Figma node-id и картами зависимостей: `docs/plans/PAGE_QUEUE.md`.

## Последствия

**Положительные:**
- рабочий пользовательский путь появляется раньше — после реализации первых 2–3 страниц
- нет риска реализовать компоненты, которые не войдут в финальный объём
- каждый компонент сразу проверяется в контексте реальной страницы, а не изолированно
- объём чётко ограничен: 28 единиц вместо неопределённых 56+

**Отрицательные / компромиссы:**
- при переходе к следующей странице может потребоваться доработка уже реализованных компонентов (если новая страница предъявляет новые требования к существующему компоненту)
- компоненты, нужные на нескольких страницах, реализуются на первой из них — нужно следить за переиспользованием

---

## Альтернативы, которые рассматривались

| Альтернатива | Почему отклонена |
|---|---|
| UI Kit First (все 56 компонентов, потом страницы) | Откладывает рабочий результат; риск реализации компонентов вне объёма |
| Параллельная реализация (компоненты и страницы одновременно) | Сложнее координировать зависимости; риск блокировок |
