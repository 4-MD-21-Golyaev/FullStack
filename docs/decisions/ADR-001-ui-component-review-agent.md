# ADR-001: Post-implementation review agent for UI components

**Статус:** Принято
**Дата:** 2026-03-21

---

## Контекст

В проекте используется библиотека UI-компонентов (`src/shared/ui/`) с жёсткими правилами композиции:
- компоненты строятся из существующих примитивов, не из raw HTML
- все CSS-значения выражаются через токены, без хардкода
- каждый компонент размещается в строго определённой группе
- все компоненты экспортируются из единственного barrel-файла

Компоненты реализуются LLM-агентами по дизайну из Figma. В ходе разработки была выявлена проблема: агенты нарушали правила композиции даже при наличии документации — использовали `<button>`, `<input>` вместо существующих компонентов, хардкодили CSS-значения, пропускали экспорт из barrel. Нарушения обнаруживались вручную при ревью, что означало возврат к исправлению уже написанного кода.

### Почему правила в документации недостаточны

Пассивная документация (CLAUDE.md, ARCHITECTURE.md) задаёт стандарт, но не гарантирует его выполнение. Агент, реализующий компонент, работает в режиме "написать код" и может не перечитывать каждое правило перед каждым решением. Самопроверка в конце того же промпта частично помогает, но агент остаётся в том же контексте и склонен принимать свои решения как верные.

---

## Решение

После каждой реализации UI-компонента запускается **отдельный review-агент** с чистым контекстом.

Review-агент не знает, как компонент был реализован, — он только читает результат и сверяет с чеклистом из шести бинарных проверок:

1. **RAW HTML** — запрещённые HTML-теги в исходнике
2. **BARREL** — компонент экспортируется из `src/shared/ui/index.ts`
3. **GROUP** — файл находится в правильной группе (`buttons/`, `inputs/` и т.д.)
4. **TOKENS** — отсутствие хардкодированных значений в CSS Module
5. **IMPORTS** — cross-group импорты используют относительные пути, не barrel
6. **TYPESCRIPT** — `npx tsc --noEmit` проходит без ошибок

Каждая проверка возвращает `PASS` или `FAIL` с доказательством (строка кода, имя файла). При `FAIL` агент обязан исправить нарушение и повторить проверку.

### Почему отдельный агент, а не самопроверка

Отдельный агент имеет чистый контекст: он не вложен в процесс реализации и не склонен интерпретировать пограничные случаи в пользу уже написанного кода. Проверки бинарны и основаны на чтении файлов, а не на субъективной оценке — это снижает зависимость от "добросовестности" агента.

---

## Последствия

**Положительные:**
- нарушения правил обнаруживаются до того, как компонент используется в других слоях
- правила перестают быть пассивной документацией — они верифицируются при каждой реализации
- review-агент служит живым документом того, что считается корректной реализацией

**Отрицательные / компромиссы:**
- каждая реализация требует двух агентных вызовов вместо одного
- review добавляет время на реализацию одного компонента
- чеклист нужно обновлять при добавлении новых правил

## Альтернативы, которые рассматривались

| Альтернатива | Почему отклонена |
|---|---|
| Только документация (CLAUDE.md) | Пассивна — не гарантирует выполнение |
| Self-review в конце того же промпта | Агент проверяет сам себя в том же контексте, склонен к confirmation bias |
| Git hook + ESLint/TSC | Ловит только синтаксические ошибки, не семантические (raw HTML, неверная группа) |
| Комбинация hook + self-review | Ловит больше, но без независимой проверки |

Выбрана комбинация **hook (tsc + eslint)** для автоматических проверок при каждом сохранении + **отдельный review-агент** для семантических проверок после реализации.
