# PLAN_PENDING_PAYMENT_RECOVERY

## Проблема

В сценарии инициации оплаты локальная запись платежа со статусом `PENDING` создается до обращения к внешнему платежному шлюзу. Если внешний вызов завершается ошибкой, в системе может остаться зависший `PENDING`-платеж без корректного завершения жизненного цикла.

Это создает риск, что повторная попытка оплаты будет блокироваться логикой "payment already in progress", хотя фактически внешний платеж не был успешно создан.

## Цель

Сделать инициацию оплаты устойчивой к сбоям внешнего платежного шлюза:

- после ошибки внешнего вызова система не должна навсегда оставаться в состоянии ложного `PENDING`
- повторная попытка оплаты должна быть либо безопасно разрешена, либо корректно контролируема
- жизненный цикл локального платежа должен отражать реальное состояние внешней операции

## Логика исправления

### 1. Явно определить целевой сценарий при сбое внешнего вызова

Нужно выбрать и зафиксировать поведение, когда локальный платеж уже создан, а внешний платежный запрос завершился ошибкой.

Наиболее логичный вариант:

- локальная запись не должна оставаться "активно ожидающей" бесконечно
- система должна переводить ее в состояние, которое не блокирует дальнейший корректный retry

Причина:

- без четкой модели восстановления любое следующее поведение будет неустойчивым и трудно тестируемым

### 2. Разделить два факта: "локальная попытка начата" и "внешний платеж реально создан"

Сейчас `PENDING` фактически означает и локальное ожидание, и наличие внешнего платежа. Это слишком сильная перегрузка одного состояния.

Логически нужно различать:

- локальная запись создана
- внешний платеж успешно создан и привязан через `externalId`
- внешний вызов упал, и локальная попытка должна считаться неуспешной или требующей повторения

Причина:

- одно состояние не должно скрывать разные операционные сценарии
- это упрощает retry-логику и диагностику сбоев

### 3. Добавить компенсационный сценарий после ошибки внешнего вызова

Если вызов YooKassa упал после создания локальной записи, система должна выполнить компенсирующее действие.

Возможные логические варианты:

- перевести платеж в `FAILED`
- пометить его как "не завершил создание внешнего платежа"
- разрешить повтор, если `externalId` не был установлен

Причина:

- иначе одна временная ошибка сети или провайдера приводит к долговременной блокировке оплаты

### 4. Сохранить идемпотентность повторной попытки

После исправления повторный запрос на оплату не должен создавать хаотичные дубликаты и не должен использовать "застрявшую" старую попытку без явного правила.

Нужно определить:

- когда старая попытка считается завершенной неуспешно
- когда допустимо создать новую
- нужно ли повторно использовать старую запись или всегда создавать новую попытку

Причина:

- надежность платежного флоу зависит не только от happy path, но и от контролируемого retry

### 5. Не смешивать recovery-логику с authorization

Эта задача относится не к проверке прав, а к согласованности платежного процесса.

Причина:

- ownership и role checks должны оставаться в HTTP-слое
- логика устойчивости платежного флоу должна оставаться внутри use case и слоя платежной интеграции

### 6. Зафиксировать поведение для последующих частей системы

После сбоя внешнего вызова нужно обеспечить предсказуемость для:

- route handler оплаты
- webhook-обработки
- timeout-обработчиков
- клиентского повторного запуска оплаты

Причина:

- если одно место считает платеж "застрявшим", а другое "активным", система станет противоречивой

## Ожидаемый результат

После исправления временный сбой YooKassa или сети не будет навсегда блокировать оплату заказа. Локальное состояние платежа будет корректно отражать факт неуспешной инициации, а повторный запуск сможет работать по понятным правилам.

## Критерии проверки

### Функциональные

1. Если внешний платежный шлюз успешно создает платеж, текущий успешный сценарий оплаты не ломается.
2. Если внешний вызов падает после создания локальной записи, система не оставляет платеж в ложном активном `PENDING` без recovery-логики.
3. После такого сбоя повторная попытка оплаты не блокируется навсегда старой записью.
4. Повторная попытка оплаты работает по заранее определенному и стабильному правилу.
5. Заказ не переходит в неконсистентное состояние из-за ошибки на этапе создания внешнего платежа.

### Архитектурные

1. Логика восстановления остается внутри платежного/application-слоя, а не размазывается по route handler'ам.
2. Решение не ломает разделение ответственности между use case, репозиторием и внешним шлюзом.
3. Состояние локального платежа отражает реальное состояние внешней операции более точно, чем раньше.

### По контракту API

1. Ошибка внешнего шлюза не маскируется как ошибка авторизации или ошибка отсутствия заказа.
2. Клиент получает предсказуемый ответ при неуспешной инициации оплаты.
3. После recovery клиентский повтор оплаты имеет понятный и стабильный результат.

### По надежности

1. Временная ошибка сети или провайдера не создает постоянную блокировку оплаты.
2. Повторный запуск не приводит к бесконтрольному накоплению зависших записей.
3. Система не теряет возможность восстановиться после частичного сбоя между локальной БД и внешним провайдером.

### По тестированию

1. Есть тест на happy path успешного создания внешнего платежа.
2. Есть тест на падение внешнего шлюза после создания локального платежа.
3. Есть тест, подтверждающий корректное состояние локального платежа после сбоя.
4. Есть тест на успешный повторный запуск после такого сбоя.
5. Есть тест, что неконсистентный `PENDING` не остается навсегда без предусмотренного сценария восстановления.

## Практический сценарий приемки

1. Создать заказ, готовый к оплате.
2. Замокать внешний платежный шлюз так, чтобы он падал после локального создания записи платежа.
3. Выполнить инициацию оплаты.
4. Убедиться, что локальная запись платежа не остается в состоянии, навсегда блокирующем повтор.
5. Повторить инициацию оплаты после сбоя.
6. Убедиться, что повторная попытка работает по ожидаемому правилу и не упирается в ложный "payment already in progress".
