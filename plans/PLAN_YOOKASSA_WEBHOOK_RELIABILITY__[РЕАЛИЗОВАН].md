# PLAN_YOOKASSA_WEBHOOK_RELIABILITY

## Проблема

Webhook YooKassa сейчас всегда отвечает `200 OK`, даже если внутренняя обработка завершилась ошибкой. Это защищает от избыточных повторов со стороны провайдера, но создает обратный риск: внутренний сбой может потерять значимое событие без механизма восстановления.

## Цель

Сделать обработку webhook-событий надежнее без потери управляемости:

- входящее событие не должно теряться из-за временной внутренней ошибки
- система должна иметь след или журнал факта получения webhook
- должна существовать стратегия восстановления или повторной обработки проблемного события

## Логика исправления

### 1. Зафиксировать целевую модель надежности webhook

Нужно определить, что считается успешной обработкой входящего webhook-события:

- событие принято системой
- событие сохранено или зааудировано
- событие обработано бизнес-логикой или поставлено на последующее восстановление

Причина:

- сейчас "ответили 200" не эквивалентно "событие надежно обработано"

### 2. Разделить прием webhook и применение бизнес-эффекта

Прием внешнего уведомления и изменение внутреннего состояния не обязаны быть одним неразделимым шагом на уровне runtime.

Логически стоит выделить:

- надежный прием события
- последующую обработку события

Причина:

- это уменьшает риск потери данных при падении между `req.json()` и завершением use case
- упрощает повторную обработку и аудит

### 3. Добавить аудит или persisted inbox для входящих webhook

Система должна иметь надежно сохраняемый след того, что событие пришло от YooKassa.

Варианты логики:

- таблица входящих webhook-событий
- persisted inbox
- журнал с состоянием "received / processed / failed"

Причина:

- без этого невозможно уверенно восстановить пропущенный платежный event

### 4. Сделать ошибки внутренней обработки наблюдаемыми и восстанавливаемыми

Если `ConfirmPaymentUseCase` падает, должно быть понятно:

- какое событие упало
- почему оно упало
- может ли оно быть повторно обработано

Причина:

- просто записать `console.error` недостаточно для надежной интеграции оплаты

### 5. Сохранить защиту от избыточных повторов со стороны провайдера

Надежность не должна сломать уже существующую идею идемпотентной обработки.

Причина:

- webhook может прийти повторно
- система должна уметь безопасно принимать повторные уведомления без дублей бизнес-эффекта

### 6. Согласовать новую модель с платежной идемпотентностью

После улучшения webhook-контур должен продолжать корректно работать с already-processed платежами.

Причина:

- если аудит входящих событий появится отдельно, его нельзя сделать конфликтующим с текущей идемпотентностью `ConfirmPaymentUseCase`

## Ожидаемый результат

После исправления внутренний сбой в момент обработки webhook больше не означает бесследную потерю события. У системы есть наблюдаемый и восстанавливаемый след получения webhook, а повторная обработка проблемных событий становится управляемой.

## Критерии проверки

### Функциональные

1. Валидный webhook по-прежнему может успешно доводить платеж до нужного бизнес-эффекта.
2. Повторный webhook для уже обработанного платежа не создает дубль состояния.
3. При внутренней ошибке факт получения webhook не теряется бесследно.
4. Проблемное webhook-событие можно идентифицировать и повторно обработать по предусмотренному механизму.

### Архитектурные

1. Прием внешнего webhook и внутренняя бизнес-обработка логически разделены.
2. Решение не ломает текущую идемпотентность платежного use case.
3. Надежность webhook строится не на одном `console.error`, а на сохраняемом следе/статусе обработки.

### По надежности

1. Временный внутренний сбой не ведет к бесследной потере события оплаты.
2. Система поддерживает восстановление или повторную обработку проблемного webhook.
3. Повторные уведомления от YooKassa остаются безопасными.

### По наблюдаемости

1. Можно определить, что webhook был получен.
2. Можно определить, был ли он обработан успешно, ошибочно или ожидает повторной обработки.
3. По логам или хранилищу можно найти причину сбоя конкретного события.

### По тестированию

1. Есть тест на успешную обработку валидного webhook.
2. Есть тест на повторный webhook для уже обработанного платежа.
3. Есть тест на внутренний сбой с сохранением аудита/входящего события.
4. Есть тест на последующую повторную обработку проблемного события.

## Практический сценарий приемки

1. Смоделировать входящий webhook `payment.succeeded`.
2. Принудительно вызвать внутреннюю ошибку в `ConfirmPaymentUseCase`.
3. Убедиться, что событие не теряется бесследно и фиксируется в предусмотренном механизме аудита/inbox.
4. Снять искусственную ошибку.
5. Запустить повторную обработку.
6. Убедиться, что платеж корректно доводится до итогового состояния без дублей.
