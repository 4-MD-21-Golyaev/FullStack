# PLAN_TEST_PAGES_ADMIN_COURIER_MVP

Status: РЕАЛИЗОВАН  
Priority: P1  
Почему: Нужен быстрый ручной контур приемки операционных сценариев без роли `STAFF`.

---

## 1) Summary

Подготовить простые сценарные тестовые страницы, где активные роли только `CUSTOMER`, `PICKER`, `COURIER`, `ADMIN`.  
Роль `STAFF` полностью исключается из логики сценариев, матрицы доступа и проверок.

## 2) Логика страниц (с учетом удаления `STAFF`)

### 2.1 `/test-ops` (хаб)

- Выбор роли: только `CUSTOMER` / `PICKER` / `COURIER` / `ADMIN`.
- Ссылки на сценарии и общий чек-лист инвариантов.
- Явный блок: `STAFF` не поддерживается.

### 2.2 `/test-picker`

Потоки: `available`, `me`, `claim/release`, `start-picking`, `complete-picking`.

Проверки:

1. `PICKER` берет заказ из `available`.
2. Повторный `claim` тем же `PICKER` идемпотентен.
3. Конкурентный `claim` другим пользователем возвращает `409`.
4. Claim невозможен вне `CREATED/PICKING`.
5. Любая попытка входа в picker-контур нецелевой ролью блокируется.

### 2.3 `/test-courier`

Потоки: `available`, `me`, `claim/release` (A: `DELIVERY`; B: расширенные статусы).

Проверки:

1. `COURIER` может claim/release только в разрешенном окне.
2. Повторный `claim` идемпотентен.
3. Параллельный `claim` дает `409`.
4. Нецелевые роли не выполняют courier-действия.

### 2.4 `/test-admin-ops`

Вкладки:

- `Orders`: реестр, фильтры, маркеры операционного состояния.
- `Payments Issues`: `retry`, `mark-failed` с обязательным `reason`.
- `Jobs`: `run/status` чувствительных job.
- `Audit`: наличие обязательных полей и событий.

Проверка: override и чувствительные операции доступны только `ADMIN`.

### 2.5 `/test-access-matrix`

- Матрица доступа только по 4 ролям: `CUSTOMER`, `PICKER`, `COURIER`, `ADMIN`.
- Отдельный контроль отсутствия `STAFF` в доступных ролях и сценариях.

## 3) Контрактные изменения для тестовых страниц

- Во всех тестовых UI/сценариях удалить упоминания `STAFF`.
- Во всех ролевых селекторах, подсказках, таблицах доступа и expected-results оставить только целевые роли.
- Проверки совместимости "`STAFF` как `PICKER`" исключить полностью.

## 4) Test Plan (ключевые сценарии)

1. `CUSTOMER` изолирован от `/api/admin/*`, `/api/picker/*`, `/api/courier/*`.
2. `PICKER` проходит полный picking-сценарий и не имеет courier/admin-прав.
3. `COURIER` проходит claim/release доставки и не имеет picker/admin-прав.
4. `ADMIN` имеет полный операционный доступ: реестр, проблемные оплаты, job, аудит.
5. Конкурентные claim-сценарии устойчивы: один успешный claim и `409` у конкурента.
6. Аудит фиксирует все критичные действия и неуспешные попытки.
7. Ни в одном тестовом сценарии, экране и матрице роль `STAFF` не фигурирует.

## 5) Assumptions

- Удаление `STAFF` уже завершено в домене, JWT/session/RBAC и API-контрактах.
- Тестовые страницы нужны для ручной приемки логики, а не для продуктового UI.
