# PLAN_PAYMENT_AUTHORIZATION_FIX

## Проблема

Endpoint `/api/orders/[id]/pay` сейчас не проверяет, что пользователь с ролью `CUSTOMER` является владельцем заказа. Это создает пробел в авторизации на уровне конкретного ресурса: аутентифицированный клиент теоретически может попытаться инициировать оплату чужого заказа, если знает `orderId`.

## Цель

Сделать поведение endpoint безопасным и согласованным с остальными маршрутами заказов:

- `CUSTOMER` может инициировать оплату только своего заказа.
- `STAFF` и `ADMIN` сохраняют доступ независимо от владельца заказа.
- Неавторизованный запрос получает `401`.
- Запрос к несуществующему заказу получает `404`.
- Проверка прав выполняется до запуска бизнес-логики оплаты и до обращения к внешнему платежному шлюзу.

## Логика исправления

### 1. Зафиксировать модель доступа для `/api/orders/[id]/pay`

Нужно явно определить ожидаемое правило доступа:

- если пользователь не аутентифицирован, вернуть `401`
- если заказ не найден, вернуть `404`
- если роль пользователя `CUSTOMER` и `order.userId !== currentUserId`, вернуть `403`
- если роль пользователя `STAFF` или `ADMIN`, разрешить выполнение
- если роль пользователя `CUSTOMER` и заказ принадлежит ему, разрешить выполнение

Причина:
Без явной модели доступа легко смешать ошибки авторизации, отсутствия ресурса и бизнес-ограничений оплаты.

### 2. Выполнять ownership-check в route handler

Проверку владельца нужно разместить в HTTP-слое, внутри route handler оплаты, до вызова `InitiatePaymentUseCase`.

Причина:

- именно route handler уже знает текущего пользователя через `x-user-id` и `x-user-role`
- это boundary-уровень, где и должна решаться авторизация доступа к конкретному ресурсу
- use case оплаты должен оставаться независимым от HTTP-контекста

### 3. Переиспользовать существующий паттерн order authorization

Логику доступа для `/pay` нужно выровнять с уже существующим поведением маршрутов просмотра и отмены заказа.

Причина:

- единая модель безопасности для операций над заказом
- меньше расхождений между endpoint'ами
- ниже стоимость сопровождения

### 4. Не переносить ownership-проверку в `proxy.ts`

`proxy.ts` должен оставаться слоем грубой аутентификации и role-based gating, а не местом для загрузки заказа из БД.

Причина:

- ownership-check требует чтения конкретного заказа
- это усложнит middleware и смешает уровни ответственности
- current design проекта уже использует route-level проверки для resource-based access control

### 5. Оставить бизнес-валидацию оплаты в use case

Проверка прав отвечает на вопрос "кто может инициировать оплату", а `InitiatePaymentUseCase` должен продолжать отвечать на вопрос "разрешена ли оплата по текущему состоянию заказа".

Причина:

- разделение обязанностей остается чистым
- авторизация не смешивается с доменной логикой
- меньше шанс на побочные регрессии в order/payment flow

### 6. Сохранить корректный контракт ответов API

После исправления endpoint должен различать причины отказа:

- `401` для отсутствующей аутентификации
- `403` для запрета доступа к чужому заказу
- `404` для отсутствующего заказа
- `400` или иное бизнес-ошибочное состояние только для валидных, но недопустимых по бизнес-логике сценариев оплаты

Причина:

- предсказуемость API
- простота клиентской обработки
- упрощение тестирования и отладки

## Ожидаемый результат

После внедрения изменений пользователь-клиент не сможет инициировать оплату чужого заказа даже при наличии корректного `orderId`. При этом административные роли не потеряют доступ, а бизнес-логика оплаты останется изолированной внутри use case.

## Критерии проверки

### Функциональные

1. Неаутентифицированный запрос на `/api/orders/[id]/pay` получает `401`.
2. Аутентифицированный `CUSTOMER` может инициировать оплату собственного заказа.
3. Аутентифицированный `CUSTOMER` получает `403` при попытке инициировать оплату чужого заказа.
4. Если заказ с указанным `id` не существует, endpoint возвращает `404`.
5. `STAFF` может инициировать оплату заказа независимо от владельца.
6. `ADMIN` может инициировать оплату заказа независимо от владельца.
7. При успешной авторизации route handler передает выполнение в `InitiatePaymentUseCase`.

### Архитектурные

1. Проверка владельца реализована в HTTP-слое, а не в доменном слое.
2. `InitiatePaymentUseCase` не начинает зависеть от HTTP-заголовков, cookies или роли пользователя.
3. Логика доступа на `/pay` согласована по смыслу с маршрутами просмотра и отмены заказа.
4. `proxy.ts` не берет на себя чтение заказа из БД ради ownership-check.

### По контракту API

1. `401` используется только для отсутствующей или невалидной аутентификации.
2. `403` используется только в случае, когда заказ существует, пользователь аутентифицирован, но не имеет права инициировать оплату.
3. `404` используется только когда заказ не найден.
4. Бизнес-ошибки оплаты не маскируются под ошибки авторизации.

### По безопасности

1. Знание `orderId` само по себе не позволяет клиенту оплатить чужой заказ.
2. Проверка прав происходит до создания локального платежа.
3. Проверка прав происходит до обращения к YooKassa или другому внешнему платежному шлюзу.

### По тестированию

1. Есть тест на `401` для неаутентифицированного запроса.
2. Есть тест на успешную инициацию оплаты владельцем заказа.
3. Есть тест на `403` для клиента, пытающегося оплатить чужой заказ.
4. Есть тест на `404` для отсутствующего заказа.
5. Есть тест на отсутствие блокировки для `STAFF` и/или `ADMIN`.
6. Есть проверка, что при `403` бизнес-логика оплаты не запускается.

## Практический сценарий приемки

1. Создать двух пользователей с ролью `CUSTOMER`: `A` и `B`.
2. Создать заказ, принадлежащий `A`.
3. Выполнить запрос на `/api/orders/[id]/pay` от имени `B`.
4. Убедиться, что ответ равен `403`.
5. Убедиться, что локальная запись платежа не создается.
6. Убедиться, что внешний платежный шлюз не вызывается.
7. Выполнить тот же запрос от имени `A` и убедиться, что запрос проходит к бизнес-логике оплаты.
