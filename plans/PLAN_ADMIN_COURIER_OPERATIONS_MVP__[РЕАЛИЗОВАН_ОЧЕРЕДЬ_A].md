# PLAN_ADMIN_COURIER_OPERATIONS_MVP

## 1) Контекст и целевая модель

Текущая backend-логика заказа уже зрелая, но операционный контур персонала неполный. Целевое состояние:

- `ADMIN` контролирует процесс и делает только аварийные вмешательства
- `PICKER` работает через self-claim в контуре сборки
- `COURIER` работает через self-claim в контуре доставки
- критичные действия диагностируются через аудит

Роли целевой модели:

- `CUSTOMER` — клиентский контур
- `PICKER` — сборка
- `COURIER` — доставка
- `ADMIN` — контроль, аварийные override, проблемные оплаты, чувствительные job

## 2) Важно: scope и этапность без скрытых рисков

План разделен на 2 очереди, чтобы не смешивать быстрый admin MVP и глубокий редизайн state machine.

### 2.1 Очередь A (Admin MVP без смены state machine)

Реализуется первой и не ломает текущий жизненный цикл `CREATED -> PICKING -> PAYMENT -> DELIVERY -> CLOSED`.

Состав:

1. admin-реестр заказов (фильтры + пагинация + операционные маркеры)
2. self-claim сборки (`PICKER`) и self-claim доставки (`COURIER`) поверх текущих статусов
3. аудит бизнес-действий и неуспешных попыток
4. безопасный контур проблемных оплат (`retry`, `mark failed` с reason)
5. job run/status для операционного запуска и диагностики

### 2.2 Очередь B (Courier Domain v2, отдельный релиз)

Выполняется только после стабилизации очереди A.

Состав:

- расширение state machine доставки:
  `... -> DELIVERY_ASSIGNED -> OUT_FOR_DELIVERY -> DELIVERED -> CLOSED`
- переразделение outbox-событий:
  `ORDER_READY_FOR_DELIVERY` после оплаты, `ORDER_DELIVERED` после факта вручения
- SLA-диагностика доставки как отдельный доменный слой

## 3) Breaking changes и совместимость

### 3.1 Роли (`STAFF -> PICKER/COURIER`)

Это breaking change. Для совместимости внедряется в два шага:

1. Шаг совместимости:
   - в JWT/session/RBAC временно поддерживаются `STAFF`, `PICKER`, `COURIER`
   - `STAFF` трактуется как `PICKER` на переходный период
2. Финальный шаг:
   - миграция пользователей `STAFF` в целевые роли
   - удаление `STAFF` из бизнес-проверок и тестов

### 3.2 Статусы заказа (только для очереди B)

Расширение статусов — отдельная миграция с обратной совместимостью API:

- до включения B API продолжает возвращать текущие статусы
- после включения B старые интеграционные точки получают маппинг/адаптацию
- обновление документации и контрактов фиксируется как отдельный acceptance gate

## 4) Бизнес-правила и инварианты

### 4.1 Матрица ответственности

- `PICKER`: видит очередь сборки и свои claimed-заказы, выполняет только свои picking-действия
- `COURIER`: видит очередь доставки и свои claimed-заказы, выполняет только свои delivery-действия
- `ADMIN`: полный обзор, override claim только с reason, проблемные оплаты, чувствительные job
- `CUSTOMER`: нет доступа к `/api/admin/*`, `/api/picker/*`, `/api/courier/*`

### 4.2 Claim инварианты

1. Не более одного активного `pickerClaim` на заказ
2. Не более одного активного `deliveryClaim` на заказ
3. Окна доступности claim фиксированы по статусам:
   - `pickerClaim`: только в `CREATED`/`PICKING`
   - `deliveryClaim` (очередь A): только в `DELIVERY`
   - `deliveryClaim` (очередь B): только в `DELIVERY_ASSIGNED`/`OUT_FOR_DELIVERY`
4. Release доступен только в соответствующем рабочем окне claim; при выходе заказа из окна claim снимается автоматически (или переводится в неактивный history-record)
5. Override только у `ADMIN` и только с обязательным `reason`

### 4.3 DB-гарантии против гонок (обязательно)

Для claim-логики обязательны не только проверки в коде, но и ограничения БД:

- атомарный `claim if null` в транзакции
- блокировки/условные update на уровне строки заказа
- уникальные инварианты на активные claim (через partial unique индекс или эквивалент)
- конфликт параллельного claim возвращает `409`, а не приводит к двойному назначению

## 5) Логика по подсистемам

### 5.1 Admin-реестр

- `GET /api/admin/orders` с фильтрами по статусу, датам, поиску (`id/email/phone`)
- пагинация (`limit/offset`)
- операционные поля: `timeInState`, `isPaymentOverdue`, `hasPendingPayment`

### 5.2 Picker/Courier self-claim

- очереди `available` и списки `me`
- claim/release с идемпотентностью для повторного claim тем же пользователем
- override release/transfer только для `ADMIN` с аудитом

### 5.3 Проблемные оплаты

- просмотр истории и текущего состояния попыток
- `retry` и `mark failed` (reason обязателен)
- без force-success/force-cancel в MVP
- без обхода доменных и транзакционных инвариантов
- `mark failed` допустим только для `PENDING`-попытки с повторной проверкой состояния в транзакции; операция не должна напрямую форсировать переход заказа в terminal state
- гонка `mark-failed` vs webhook обрабатывается детерминированно: приоритет `SUCCESS` при коммите первым, проигравшая операция становится no-op/идемпотентным ответом

### 5.4 Job run/status

- запуск job по ролям
- статус последнего запуска с `startedAt/finishedAt`, инициатором, счетчиками и `error summary`
- чувствительные job ограничиваются ролью `ADMIN`
- admin-эндпоинты job являются фасадом: они не открывают `/api/internal/jobs/*` наружу и не ослабляют существующую secret-only защиту внутренних job-роутов

### 5.5 Аудит

Обязательные события:

- claim/release/override
- start/complete picking
- delivery-действия (для очереди B)
- retry/mark-failed оплаты
- run job
- failed attempts критичных операций

Обязательные поля:

- `actorUserId`, `actorRole`
- `action`, `targetType`, `targetId`
- `before`, `after`
- `reason` (где обязателен)
- `timestamp`
- `correlationId`

## 6) API-контур

### 6.1 Очередь A (реализуется первой)

- `GET /api/admin/orders`
- `GET /api/picker/orders/available`
- `GET /api/picker/orders/me`
- `POST /api/picker/orders/:id/claim`
- `POST /api/picker/orders/:id/release`
- `GET /api/courier/orders/available`
- `GET /api/courier/orders/me`
- `POST /api/courier/orders/:id/claim`
- `POST /api/courier/orders/:id/release`
- `GET /api/admin/payments/issues`
- `POST /api/admin/payments/:id/retry`
- `POST /api/admin/payments/:id/mark-failed`
- `POST /api/admin/jobs/:jobName/run`
- `GET /api/admin/jobs/:jobName/status`
- `/api/internal/jobs/*` остаются недоступными по JWT и вызываются только из доверенного контура (secret-only)

### 6.2 Очередь B (второй релиз)

- `POST /api/courier/orders/:id/start-delivery`
- `POST /api/courier/orders/:id/confirm-delivered`
- `POST /api/courier/orders/:id/mark-delivery-failed`
- расширенные admin override endpoints по claim, если не вошли в A

## 7) Критерии приемки

### 7.1 Очередь A

1. CUSTOMER полностью изолирован от операционных API
2. Параллельные claim не создают двойных назначений, конфликт отдается как `409`
3. Проблемные оплаты обрабатываются только через безопасные операции
4. Job run/status работает по ролевой модели
5. Все критичные действия и ошибки отражены в аудите
6. Claim невозможен вне разрешенных окон статусов (`pickerClaim`: `CREATED`/`PICKING`, `deliveryClaim`: `DELIVERY` для очереди A)
7. Переходный `STAFF` корректно работает в picking-контуре как `PICKER` (нет регрессий 403/401 для этого контура до финальной миграции ролей)
8. Гонки `mark-failed`/webhook не нарушают инварианты оплаты и статуса заказа
9. Существующие тесты остаются зелеными, build проходит

### 7.2 Очередь B

1. `CLOSED` недоступен до фактического `DELIVERED`
2. `ORDER_DELIVERED` публикуется только после подтвержденного вручения
3. Новые статусы доставки не ломают совместимость API/интеграций
4. Регрессия заказа/оплаты/outbox остается зеленой

## 8) Практический приемочный сценарий

### Сценарий для очереди A

1. Пользователь с переходной ролью `STAFF` корректно работает как `PICKER`
2. `PICKER` берет заказ из `available`, выполняет сборку, завершает `PICKING`
3. `COURIER` берет заказ в доставку через claim/release контур
4. `ADMIN` обрабатывает проблемный платеж через `retry` или `mark-failed` с reason
5. `ADMIN` запускает job и получает статус выполнения
6. По всем операциям есть audit-записи с корректным `correlationId`

### Сценарий для очереди B

1. Заказ проходит `DELIVERY_ASSIGNED -> OUT_FOR_DELIVERY -> DELIVERED`
2. `CLOSED` разрешается только после `DELIVERED`
3. Событие `ORDER_DELIVERED` в outbox формируется только после факта вручения

## 9) Матрица рисков внедрения

| Риск | Влияние | Вероятность | Митигация | Критерий контроля |
|---|---|---|---|---|
| Ломающее изменение ролей (`STAFF -> PICKER/COURIER`) | Высокое | Средняя | Двухшаговая совместимость: временная поддержка `STAFF` в JWT/RBAC, затем финальная миграция | Нет 403/401 регрессий для бывших `STAFF`; все тесты RBAC зеленые |
| Регрессии из-за расширения state machine (очередь B) | Высокое | Средняя | Вынос в отдельный релиз, feature flag, контрактные тесты старых/новых статусов | Старые интеграции продолжают работать до включения B; после включения есть маппинг |
| Гонки при self-claim | Высокое | Средняя | Транзакционный `claim if null`, row-level lock/условный update, DB-инварианты | Параллельный claim дает одного победителя и `409` для конкурента |
| Неполный аудит критичных действий | Среднее | Средняя | Единый слой audit-записи для всех операционных use case | 100% критичных операций из списка попадают в аудит с `correlationId` |
| Непредсказуемое поведение проблемных оплат | Высокое | Низкая-Средняя | Ограничить набор операций (`retry`, `mark failed`), запретить force-операции, проводить через доменные инварианты | Нет обхода state machine; повторные вызовы идемпотентны |
| Параллельные ручные/cron job приводят к дублированию | Среднее | Средняя | Режимы запуска через общий lock/claim-механизм, статус последнего запуска | Нет дублей обработки одного и того же элемента; статусы запусков консистентны |
| Рост сложности поддержки после очереди B | Среднее | Средняя | Четкая граница A/B, обновленные контракты, отдельные acceptance gates | Реализация B принимается только при нулевой регрессии очереди A |
